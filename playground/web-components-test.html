Web Components – Shadow DOM + Teleport

Demonstrates that GModal, GAlertDialog, and GPopover work correctly as Web Components (custom elements) without any extra props. Slotted content is visible and overlay stacking is correct because Teleport automatically targets the shadow root when inside a custom element.

How it works

By default these components teleport their content to #modal-root in the light DOM. Inside a Web Component, this would move the native <slot> out of the shadow DOM, orphaning any slotted content.

The fix: each component calls (getCurrentInstance() as any)?.ce?.shadowRoot during setup. When running inside a defineCustomElement-based custom element, instance.ce is the host element and instance.ce.shadowRoot is its shadow root. That shadow root is used as the Teleport target instead of #modal-root. This keeps the <slot> inside the shadow DOM so slotted content is correctly distributed, while still using position: fixed and the shared z-index stack for correct overlay ordering. 

// In each component (GModal / GAlertDialog / GPopover):
const _ce = (getCurrentInstance() as any)?.ce as Element | undefined;
const teleportTarget = _ce?.shadowRoot ?? '#modal-root';

In a regular Vue.js app, instance.ce is undefined so the target falls back to #modal-root — the original behavior is unchanged.

The no-teleport prop (edge-case opt-out)

The no-teleport prop is still available as an escape hatch for cases where even shadow-root teleport isn't appropriate. When set, the modal renders fully inline (no Teleport at all).

Usage in a Vue.js App

Regular Vue.js app behavior is completely unchanged. When not inside a custom element, instance.ce is undefined, so the Teleport target falls back to #modal-root exactly as before. 

<!-- Vue.js app – unchanged, no new props needed -->
<GModal label="My Modal">
    <p>Content here</p>
</GModal>

<!-- Web Component – also works with no extra props -->
<g-modal label="My Modal">
    <p>Content here</p>
</g-modal>
