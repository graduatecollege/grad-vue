Web Components – no-teleport Test

Demonstrates the no-teleport prop that fixes slotted content in Shadow DOM / Web Component contexts.

The Problem

Without no-teleport, the modal's internal DOM is teleported out to #modal-root in the light DOM. Slotted content stays inside the custom element's shadow DOM and is never rendered because there is no longer a <slot> inside the shadow DOM.

Click the button below. The modal opens but shows no content (only the header/close button) because the slot is orphaned.

This content should appear inside the modal, but it won't because the slot was teleported away.

The Fix – no-teleport

Adding no-teleport (or :no-teleport="true" in a Vue template) disables the teleport so that the modal renders inline, keeping the <slot> inside the shadow DOM where the slotted content can be distributed into it.

Click the button below. The modal opens with its slotted content visible.

Hello sir! This slotted content is visible because no-teleport keeps the modal's DOM (and its <slot>) inside the shadow root.

  • Close button works ✓
  • Escape key closes the modal ✓

Usage in a Vue.js App

In a regular Vue.js application the behavior is unchanged. Teleport continues to work normally by default. The no-teleport prop only needs to be added when using the components as Web Components (custom elements) where shadow DOM prevents teleport from working with slotted content. 

<!-- Vue.js app – works as before (no change needed) -->
<GModal label="My Modal">
    <p>Content here</p>
</GModal>

<!-- Web Component / custom element context -->
<g-modal label="My Modal" no-teleport>
    <p>Content here</p>
</g-modal>
