Web Components – Teleport + Overlay Stacking

All overlay components (GModal, GAlertDialog, GPopover) teleport to #modal-root in the light DOM. This keeps every overlay as a sibling in one shared container, so z-index stacking works correctly across the whole page — even when the components are used as custom elements.

How it works

All three components always teleport to #modal-root in the light DOM. When used as a Web Component, slotted content would break if the native <slot> element were moved out of the shadow DOM.

The fix: instead of a native <slot> element inside the teleported container, the components call $slots.default?.() directly via a functional component. Vue stores the light DOM children of <g-modal> as vnodes in $slots.default regardless of where the render happens, so the content is correctly rendered inside #modal-root. 

// Inside the teleported container (GModal / GAlertDialog / GPopover):
// Before (broken with web components):
//   <slot />  ← becomes native <slot> which must stay in shadow DOM

// After (works everywhere):
//   <component :is="() => $slots.default?.()" />  ← renders vnodes directly

In a regular Vue app, <component :is="() => $slots.default?.()"/> is functionally equivalent to <slot />. No API change is needed for Vue consumers.

The no-teleport prop (escape hatch)

If #modal-root is unavailable or you explicitly want inline rendering, set no-teleport. The component renders inline (inside the shadow DOM). Note that when components are not co-located in #modal-root, overlay stacking between different web components depends on the host elements not creating stacking contexts.

Usage in a regular Vue.js app

Behavior is completely unchanged. In a regular Vue app, <component :is="() => $slots.default?.()"/> is equivalent to <slot />. No API changes needed. 

<!-- Vue.js app – unchanged -->
<GModal label="My Modal">
    <p>Content here</p>
</GModal>

<!-- Web Component – now works without extra props -->
<g-modal label="My Modal">
    <p>Content here</p>
</g-modal>
